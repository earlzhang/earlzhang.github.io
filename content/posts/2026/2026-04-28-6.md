[CodeBuddy](https://www.codebuddy.cn/ide/)，腾讯家Agent工具。此前因为Deepseek V4官方发布微信中[意外的提及](https://mp.weixin.qq.com/s/8bxXqS2R8Fx5-1TLDBiEDg)，关注度又高了一把。

是的，在一众编程和Agent大咖中，出现CodeBuddy是有点突兀的，以至于许多人甚至猜测这是否Deepseek的示好，甚至暗示未来的腾讯入股有希望。

![alt text](https://earlllm.oss-cn-shanghai.aliyuncs.com/20260428_204930_856_a1c10e63_tmpzq64gosd.png)

也正因为这个原因，我顺带将它家的[WorkBuddy](https://www.codebuddy.cn/work/)，一个主打办公的Agent也安装尝试了一把。

很可惜，感觉至少WorkBuddy还接不住Deepseek溢出的厚爱。

目前，WorkBuddy的内置模型，还是老样子，虽然Hy3 Preview作为亲儿子有了，但是Deepseek还是停留在V3.2时代。

![alt text](https://earlllm.oss-cn-shanghai.aliyuncs.com/20260428_204930_698_3db489cc_image-1.png)

当然，WorkBuddy还是支持自定义模型的，所以我就接上了DeepSeek的API，用上了Deepseek V4 Flash。

很可惜，很快就遇到了报错：

![alt text](https://earlllm.oss-cn-shanghai.aliyuncs.com/20260428_204930_805_7da7ebe4_tmpaibwr2wj.png)

这个错，是Deepseek V4 Flash的坑。

以下是我请Gemini帮我综述的解释：

>Bug 的核心逻辑：为什么会报错？
传统的 OpenAI 或 Anthropic 协议中，模型思考（Thinking）和执行工具（Tool Call）通常是独立的，或者在下一轮对话中不需要回传思考过程。
>
>但 DeepSeek V4 的新规定要求：
>
>必须回传（Round-trip）： 在开启“思考模式”下，如果模型在第一轮进行了工具调用，那么它生成的 reasoning_content（思维链内容）必须作为历史上下文，在第二轮请求中完整地传回给 API。
>
>后果： 如果框架在处理消息时，为了节省 Token 或遵循旧协议将 reasoning_content 字段剥离（Strip）掉，DeepSeek API 会直接返回 HTTP 400 错误：The reasoning_content in the thinking mode must be passed back to the API.

是的，这次对Deepseek V4的发布，小小的失望也在这里，官微说：

>DeepSeek-V4 针对 Claude Code 、OpenClaw、OpenCode、CodeBuddy 等主流的 Agent 产品进行了适配和优化

但这个bug，只要用Deepseek V4 Flash随便用一会儿只要调用了工具，就能出现，难道Deepseek的测试，仅限于高端的Deepseek V4 Pro？这就有点“何不食肉糜”了。

这个bug，不大。所以类似OpenClaw、Hermes之类，都是很快修复了。

OpenClaw毕竟全球用的人多，4月24日Deepseek V4发布当天，就发新版修复了。

![alt text](https://earlllm.oss-cn-shanghai.aliyuncs.com/20260428_204930_844_4dc48002_tmpe81tcayw.png)

Hermes也是差不多的速度。

![alt text](https://earlllm.oss-cn-shanghai.aliyuncs.com/20260428_204930_862_d70310bf_tmpmoky00it.png)

但是，哪怕是2026年4月28日今天使用Workbuddy，自行调用Deepseek V4 Flash，依然是讨厌的思考传入问题。

只能说，拥抱国产模型，但Agent基建也很重要，要享受快速的更新打补丁，还得用几大生态。

当然，我知道Workbuddy是主打白领的，本来就求稳，内置都还没上Deepseek V4 Flash。

不过本来觉得好歹接一波Deepseek V4传过并溢出的厚爱，只能说，是我想多了。
